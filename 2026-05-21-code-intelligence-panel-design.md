# Code Intelligence Panel — Design Spec

**Date:** 2026-05-21
**Status:** Approved
**Feature Layer:** ML/NLP Research Layer (Layer 2)

---

## Context

The AI Software Engineer platform guides users through the full SDLC: requirements → design → implementation → quality. The Quality phase (`ValidationLab`) currently auto-generates and runs tests. However, it has a blind spot: it tells you *if* code passes tests, but not *why* a function is risky, and not *whether* the code actually covers the stated requirements.

This feature adds a **Code Intelligence Panel** — a new tab inside `ValidationLab` that provides two complementary views powered by ML and semantic search:

1. **Defect Risk Predictor** — a Random Forest model trained on the PROMISE dataset (KC1, KC2, PC1) that predicts per-function defect probability and explains contributing factors via SHAP.
2. **Requirements-to-Code Traceability** — SBERT semantic embeddings that link each SRS requirement to the code functions that implement it, flagging gaps and orphaned code.

**Research contribution:** First empirical study applying classical software defect metrics (trained on human-written C code from PROMISE) to LLM-generated code. Ablation study compares RF model vs. LLM-only review vs. combined baseline.

## Component 1: ML Defect Risk Predictor

### Training Pipeline (one-time setup)

**Dataset:** NASA PROMISE repository (GitHub mirror: `klainfo/DefectData`)
- Files used: `kc1.arff`, `kc2.arff`, `pc1.arff` (~800 labeled modules)
- Label: `bug` column (0 = no defect, 1 = defect)
- Class imbalance: ~15-20% defective — use `class_weight='balanced'` in RF

**Feature vector (5 features, radon-extractable):**

| Feature | Tool | PROMISE Column |
|---------|------|----------------|
| Cyclomatic Complexity | `radon cc` | `v(g)` |
| Halstead Volume | `radon hal` | `ev(g)` |
| Halstead Effort | `radon hal` | `n` |
| Lines of Code | `radon raw` | `loc` |
| Number of functions | `radon raw` | `n` |

Only these 5 features are used — columns that radon can compute at inference time, avoiding train/inference feature mismatch.

**Model:** `RandomForestClassifier(n_estimators=100, class_weight='balanced', random_state=42)`

**Evaluation (held-out 20% test split):**
- AUC-ROC, F1, Precision, Recall
- Saved to `models/model_metadata.json`

### Inference Pipeline (per request)

1. Receive `{ code: string, language: string }` from Node proxy
2. If language == `python`: extract function bodies, run `radon cc` + `radon hal` + `radon raw` on each
3. If language == `javascript`/other: approximate CC by counting `if/for/while/switch/case/catch` branches per function; LOC by line count; Halstead by unique operator/operand count
4. Build feature vector per function → run through `defect_rf_v1.joblib`
5. `predict_proba()` gives defect probability (0.0–1.0)
6. Risk label: `High` ≥ 0.6, `Medium` 0.35–0.59, `Low` < 0.35
7. SHAP TreeExplainer: top 3 features with direction and magnitude

**Output per function:**
```json
{
  "name": "process_payment",
  "risk_score": 0.81,
  "risk_label": "High",
  "metrics": { "cc": 14, "halstead_volume": 892, "loc": 67 },
  "shap_explanation": [
    "High cyclomatic complexity (+0.34 risk)",
    "High Halstead effort (+0.21 risk)",
    "Low LOC relative to complexity (+0.14 risk)"
  ]
}
```

## Component 2: Requirements-to-Code Traceability

### Model

`sentence-transformers/all-MiniLM-L6-v2` (80MB, downloaded once to `model_cache/` by `download_models.sh`). Same model planned for conflict detector and traceability — no duplicate download.

### Similarity Thresholds

| Score | Strength | Display |
|-------|----------|---------|
| ≥ 0.65 | Strong link | Green cell |
| 0.45–0.65 | Weak link | Yellow cell |
| < 0.45 | No link | Empty |

### Coverage Formula

`coverage_pct = (requirements with ≥1 strong link) / total_requirements × 100`

### Orphan Detection

- **Orphaned requirement**: no code function has similarity ≥ 0.45 — the requirement is unimplemented
- **Orphaned code**: no requirement has similarity ≥ 0.45 — the function has no stated requirement

### Input Extraction

- Requirements: extracted from project sidebar documents marked "Use in context" (same context injection pattern already in the app — parsed line-by-line, each `shall` sentence is a requirement)
- Code functions: regex-extracted from the generated code (Python `def func_name`, JS `function funcName` / `const funcName = `)

## API Contracts

### `POST /api/ml/defect/predict`

**Request:**
```json
{
  "code": "def process_payment(amount):\n    ...",
  "language": "python"
}
```

**Response (200):**
```json
{
  "functions": [
    {
      "name": "process_payment",
      "risk_score": 0.81,
      "risk_label": "High",
      "metrics": { "cc": 14, "halstead_volume": 892, "loc": 67 },
      "shap_explanation": ["High cyclomatic complexity (+0.34)", "..."]
    }
  ],
  "summary": { "high_risk": 1, "medium_risk": 2, "low_risk": 4 }
}
```

**Error (503):** `{ "error": "Defect prediction service unavailable" }` — shown as toast, does not crash the app.

---

### `POST /api/ml/traceability/analyze`

**Request:**
```json
{
  "requirements": ["The system shall authenticate users via JWT", "..."],
  "code_functions": [
    { "name": "authenticate_user", "signature": "def authenticate_user(token)", "docstring": "Validates JWT token" }
  ]
}
```

**Response (200):**
```json
{
  "matrix": [[0.82, 0.21], [0.15, 0.91]],
  "links": [
    { "req_idx": 0, "func_name": "authenticate_user", "score": 0.82, "strength": "strong" }
  ],
  "orphaned_reqs": [2],
  "orphaned_code": ["generate_report"],
  "coverage_pct": 85.7
}
```

**Error (503):** Same pattern as defect endpoint.

## Frontend: `CodeIntelligencePanel.jsx`

### Layout (inside ValidationLab "Intelligence" tab)

```
┌─────────────────────────────────────────────────────────────────┐
│  [Analyze Code Intelligence]  button                           │
│  (uses code already typed in ValidationLab's code input)       │

│  ● process_payment  81%  │       auth  pay  report             │

│    • High CC (+0.34)     │  Req 2  □□  ■■   □□                │
│    • High effort (+0.21) │  Req 3  ⚠   □□   □□  ← orphaned   │
│                          │                                      │
│  ● validate_input   42%  │  Coverage: 85.7%                    │
│    [Medium]  ████░░░░░   │  2 orphaned requirements            │
│                          │  1 orphaned code function           │
│  ● get_user     12%      │                                      │
│    [Low]     █░░░░░░░░   │                                      │
└──────────────────────────┴──────────────────────────────────────┘
```

### Behavior

- **Trigger:** "Analyze Code Intelligence" button sends one parallel call to both endpoints
- **Code source:** pre-filled from the existing code textarea in ValidationLab (same UX pattern)
- **Requirements source:** auto-extracted from sidebar context documents (same pattern as SystemDesignWizard)
- **Loading state:** spinner per panel (defect and traceability can resolve independently)
- **Empty state:** if no requirements in context → traceability panel shows "Add SRS documents to the sidebar and mark them as context to enable traceability"
- **Error state:** if ML service is down → show warning banner, rest of ValidationLab still works

## Error Handling

- ML service down → both panels show warning toast, ValidationLab's test generation still works
- No requirements in sidebar → traceability panel shows instructional empty state
- Language not Python → JS approximation mode (documented limitation in paper)
- Code has no detectable functions → defect panel shows "No functions detected — paste a complete code file"
- PROMISE download fails → `train_defect_model.py` uses a small synthetic fallback dataset (50 examples) so the model still loads

## Out of Scope

- Real-time analysis as user types (on-demand button only)
- Saving intelligence results to the database (shown in UI, downloadable as JSON)
- Supporting all languages with radon-equivalent metrics (Python full support, others approximated)
- Replacing the existing test generation in ValidationLab (this is additive)
