## Vision

### 10x Check

The 10x version of this tool is not "a better document generator" — it is **the only SDLC tool that actively learns from your project artifacts and predicts problems before they happen**. 

Imagine: you paste a project description → the system detects you're building a fintech app → pre-fills HIPAA/PCI requirements → you refine the SRS → the NLP layer flags 4 ambiguous requirements before you move on → you design the schema → the traceability matrix auto-links design entities to SRS requirements → you generate code → the defect predictor says "Module `PaymentService` has 81% defect risk due to high coupling" → you fix it → phase readiness scorer says "you're 94% ready to move to testing." 

No other BTech project in India is doing this. No other open-source tool is doing this end-to-end. This is the 10x version.

### Platonic Ideal

A senior engineer joins a project on day 1. They don't read every document sequentially — they scan for inconsistencies, ambiguities, and gaps. They ask "where's the traceability?" They flag risky modules without running the code. They know which requirements will cause implementation nightmares.

The platonic ideal of this tool is that AI assistant — always watching the entire SDLC, automatically surfacing what a senior engineer would catch. Users shouldn't have to ask "is my SRS good?" — the tool should tell them before they ask.

## Accepted Scope

### RESEARCH PILLAR 1: Requirement Intelligence
- **NLP Requirements Quality Analyzer**: spaCy + rule-based + Gemini explanation hybrid pipeline
  - Detects: vagueness (unmeasurable terms like "fast", "easy", "secure" with no metric), ambiguity (pronouns with unclear antecedents, multiple valid interpretations), incompleteness (missing actor, action, or object), conflicts
  - Pipeline: spaCy tags and parses each requirement sentence → rule-based patterns flag candidates → Gemini is called ONLY on flagged sentences to generate a plain-English explanation ("This requirement is ambiguous because 'it' could refer to the system or the user"). Gemini is NOT used for detection, only explanation.
  - Scores each requirement on a quality scale (0-100): 100=clear, measurable, unambiguous
  - Evaluation: PURE dataset (available at https://zenodo.org/record/1414117 under CC BY 4.0, 79 requirements from 5 projects)
  - New endpoint: `POST /api/ml/requirements/analyze` — input: `{requirements: string[]}`, output: `{scores: [{text, score, issues: [{type, explanation}]}]}`
  - Fallback: if ML service unavailable, endpoint returns `503` with `{error: "Analysis service unavailable", cached: lastResult | null}`
  - UI: quality score overlay on SRS editor, per-requirement badges (green ≥80 / yellow 50-79 / red <50)

### RESEARCH PILLAR 2: Defect Risk Intelligence
- **ML Defect Risk Predictor**:
  - Training: NASA PROMISE dataset (KC1, KC2, PC1, PC3, CM1 — ~800 modules with fault labels). Dataset mirror: https://github.com/klainfo/DefectData — files `kc1.arff`, `kc2.arff`, `pc1.arff`, `pc3.arff`, `cm1.arff`. Convert ARFF to CSV with `scipy.io.arff.loadarff`. The original promise.site.uottawa.ca URL is offline — use only the GitHub mirror.
  - **Feature vector (radon-extractable only)**: cyclomatic complexity (CC), Halstead volume (h_volume), Halstead effort (h_effort), LOC (lloc), number of functions per module. CBO/LCOM are excluded because radon does not produce them — training uses only the PROMISE columns that map to radon outputs to avoid train/inference feature mismatch.
  - Static analysis: radon extracts CC/Halstead/LOC from Python code; for JS/TS, ESLint rule-violation count is added as a separate auxiliary feature (not part of the PROMISE-trained vector — it is a supplemental risk signal shown separately).
  - Model: Random Forest (100 trees, primary) + Logistic Regression (baseline for paper comparison)
  - SHAP TreeExplainer for per-prediction explanations (adds ~50ms per call — acceptable for on-demand use)
  - Output: per-function defect probability (0-1) + risk label (Low/Medium/High) + top SHAP features
  - New endpoint: `POST /api/ml/defect/predict` — input: `{code: string, language: string}`, output: `{functions: [{name, risk_score, risk_label, shap_explanation}]}`
  - Fallback: if ML service unavailable, return `503` + show LLM-only review as fallback
  - Evaluation: AUC-ROC, F1, Precision/Recall on 20% held-out PROMISE test split
  - Model versioning: saved as `models/defect_rf_v1.joblib` with `models/model_metadata.json` containing `{version, features, training_date, dataset, metrics}`
  - Paper: "Defect Risk Prediction in LLM-Generated Code Using Classical Code Metrics and Random Forest"

### RESEARCH PILLAR 3: Cross-Artifact Intelligence
- **Requirements-to-Code Traceability Matrix** (req→code scope, not req→design→test — paper claim scoped accordingly):
  - Model: sentence-transformers `all-MiniLM-L6-v2` (80MB, HuggingFace). Pre-download required: `scripts/download_models.sh` runs at setup, downloads to `ml-service/model_cache/` and sets `TRANSFORMERS_CACHE` env var. No HuggingFace call at demo time.
  - Embeds: every SRS requirement sentence + every generated code function signature + docstring (first 200 chars)
  - Cosine similarity threshold: **0.65** (link accepted), **0.45-0.65** (weak link, shown in yellow), **<0.45** (no link)
  - Coverage formula per SRS section: `(requirements with ≥1 link above 0.65) / (total requirements in section) × 100%`
  - Flags: orphaned requirements (0 links ≥0.45), orphaned code (no requirement link ≥0.45)
  - New endpoint: `POST /api/ml/traceability/analyze` — input: `{requirements: string[], code_functions: [{name, signature, docstring}]}`, output: `{matrix: [[score]], coverage_by_section: {}, orphaned_reqs: [], orphaned_code: []}`
  - Fallback: 503 + user-visible warning "Traceability analysis unavailable"
  - UI: interactive matrix with click-to-highlight, coverage badge per section
  - Paper: "Automated Requirements-to-Code Traceability Using Sentence Transformers in an AI-Assisted SDLC Tool"

### RESEARCH PILLAR 4: Phase Readiness Intelligence
- **SDLC Phase Readiness Scorer**:
  - Implementation: **rule-based threshold classifier** (explicitly NOT a trained ML model — labeled training data does not exist). Uses logistic-regression-inspired weighted scoring across features. SHAP-style feature attribution computed analytically (feature contribution = weight × feature_value / total_score).
  - Labeling strategy: 50 synthetic project snapshots hand-labeled by the student as "ready" / "not ready" per gate. Labeling instrument: a JSON file `ml-service/train/readiness_snapshots.json` with schema `{phase, features: {...}, label: 0|1, notes: string}`. Sample labels for each gate:
    | Phase Gate | "ready=1" criteria | "not ready=0" criteria |
    |---|---|---|
    | REQ→DESIGN | quality_avg ≥70, req_count ≥10, nfr_present=true | quality_avg <50 OR req_count <5 |
    | DESIGN→IMPL | diagram_exists=true, schema_defined=true, design_words ≥500 | Any required artifact missing |
    | IMPL→TEST | defect_risk_avg <0.5, review_score ≥60, code_loc ≥50 | defect_risk_avg ≥0.7 OR review_score <40 |
  - These 50 snapshots are created by the student during Weeks 3-4 (background task) and used to train the logistic regression offline in `ml-service/train/train_readiness.py`, saved as `models/readiness_lr_v1.joblib`.
  - Features per phase gate:
    - REQ→DESIGN: SRS section completeness % (filled sections / total sections), average requirement quality score (from Pillar 1), total requirement count, NFR section present (binary)
    - DESIGN→IMPL: system diagram generated (binary), schema defined (binary), design doc word count ≥500 (binary)
    - IMPL→TEST: defect risk average across all functions (from Pillar 2), LLM review score (from existing ValidationLab), code line count ≥50 (binary)
  - Output: readiness score (0-100) + top 3 blocking factors with specific improvement suggestions
  - New endpoint: `GET /api/ml/readiness/:projectId/:phase` where `phase ∈ {req_to_design | design_to_impl | impl_to_test}` — returns `{score, label: "ready"|"not_ready"|"borderline", blocking_factors: [{factor, current_value, target_value, suggestion}]}`
  - Fallback: 503 + rule-based fallback using only binary feature checks (no model needed)
  - UI: Phase Gate dashboard with readiness gauge + blocking issues list

### INFRASTRUCTURE: Python ML Microservice
- **New service**: `ml-service/` FastAPI app on port **5001** (port 5000 conflicts with macOS AirPlay Receiver — use 5001)
- ML service binds to `127.0.0.1:5001` only (not `0.0.0.0`) — never directly accessible from outside. All ML calls go through the authenticated Node.js proxy (`/api/ml/*` requires active session). This means unauthenticated users cannot hit the ML service directly.
- Input size limits enforced at the FastAPI layer: max 50 requirements per analyze call, max 10 KB code string per defect-predict call, max 100 code functions per traceability call. Returns 422 on validation failure, 429 if rate limit exceeded (10 req/min per session via middleware).
- SHAP and ML models loaded at FastAPI service startup using `lifespan` event (not per-request). Do NOT run with `--reload` in demo/production mode — SHAP cold start is 2–5s per worker restart.
- Dependencies: scikit-learn, sentence-transformers, spaCy (en_core_web_sm), shap, radon, fastapi, uvicorn, pandas, numpy
- Node.js backend proxies ML requests: `http://localhost:5001`
- Models saved with versioning: `ml-service/models/defect_rf_v1.joblib`, `ml-service/models/model_metadata.json`
- Training scripts run once: `ml-service/train/train_defect.py`, `ml-service/train/train_readiness.py`
- Model pre-download script: `scripts/download_models.sh` — bash script that runs: `python -c "from sentence_transformers import SentenceTransformer; SentenceTransformer('all-MiniLM-L6-v2')"` with `TRANSFORMERS_CACHE=ml-service/model_cache` set. Script is idempotent (subsequent runs detect the cache directory and skip download). On partial failure (network cut mid-download), the incomplete cache is deleted and the script exits non-zero. Run once at dev setup before starting the ML service.
- Failure modes:
  | Endpoint | Failure | Response |
  |---|---|---|
  | /ml/requirements/analyze | ML service down | 503 + cached last result if available |
  | /ml/defect/predict | ML service down | 503 + LLM review shown as fallback |
  | /ml/traceability/analyze | ML service down | 503 + user warning |
  | /ml/readiness/:phase | ML service down | Rule-based fallback (binary checks only) |
  | Any endpoint | Input malformed | 422 with field-level validation errors |

### STATIC ANALYSIS SERVICE (supports Pillar 2)
- **New endpoint**: `POST /api/ml/static/analyze` — runs radon (Python) or counts ESLint violations (JS/TS)
  - Python: `radon cc -s -j <file>` → cyclomatic complexity per function; `radon hal <file>` → Halstead metrics; `radon mi <file>` → maintainability index
  - JS/TS: `eslint --format json` → violation count per file (used as auxiliary risk signal only, not fed into PROMISE-trained model)
  - Output schema: `{functions: [{name, cc, halstead_volume, halstead_effort, lloc, risk_label}], eslint_violations: int}`

### RESEARCH EVALUATION MODULE
- **New endpoint**: `GET /api/ml/eval/benchmark` — returns pre-computed evaluation metrics (not live computation)
- Evaluation runs offline in `ml-service/eval/evaluate_models.py` and saves results to `ml-service/eval/results.json`
- Frontend displays: static F1/AUC-ROC table for defect predictor (RF vs LR vs LLM-only baseline) + static precision/recall table for traceability
- No live Recharts interactive curves — results shown as formatted tables with a "last evaluated" timestamp

### PRACTICAL ENHANCEMENTS
- **GitHub PAT Import** (not full OAuth — scoped to reduce complexity): User pastes a GitHub Personal Access Token + repo URL. Backend uses `@octokit/rest` to fetch repo file tree and selected file contents. Imports code into Implementation Lab and README into project context. No OAuth flow, no token storage in DB (session only).
- **Real-time Collaboration** → DEFERRED (see updated Deferred list below — adds non-research complexity with no paper ROI)
- **Criteria-driven Test Case Generation**: Equivalence partitioning + boundary value analysis + decision table testing rules applied to parsed requirements. LLM generates test code; test design logic is deterministic. New endpoint: `POST /api/test/generate-from-requirements`
- **SDLC Analytics Dashboard**: Recharts — requirement quality trend over time, defect risk by session, traceability coverage %. Reads from SQLite project history.
- **Research Evaluation UI**: Static table view of model benchmark results (F1, AUC-ROC, Precision, Recall per model). Results from offline evaluation script, displayed without live computation.

## Research Paper Potential

This project supports 2 publishable papers:

**Paper 1**: "An Integrated NLP-ML Framework for Requirements Quality Assessment and Defect Risk Prediction in AI-Assisted SDLC Platforms"
- Novel: combines requirement ambiguity detection + defect risk in one SDLC tool
- Evaluation: PURE dataset (req quality) + PROMISE dataset (defect prediction)
- Baseline: LLM-only approach vs hybrid NLP+ML approach
- Venue: ICSME, ICSE NIER track, or MSR

**Paper 2**: "Automated Requirements-to-Code Traceability Using Sentence Transformers in an AI-Assisted SDLC Tool"
- Scope: req→code traceability (scoped deliberately — req→design→test is future work)
- Novel: first traceability system applied specifically to LLM-generated code artifacts
- Evaluation: sample of 20 student projects — manually verify ground truth links vs SBERT predictions (precision/recall/F1). Report threshold sensitivity (0.45, 0.55, 0.65, 0.75).
- Venue: REFSQ, NLBSE workshop at ICSE, or MSR

### Critical Bugs Pre-empted

1. **`onProxyRes` fires before body is buffered** → switched to `responseInterceptor` — without this, every SQLite write would save an empty payload
2. **`radon.h_visit` on whole file gives module-level Halstead to every function** → must call `h_visit(fn_src)` per function body extracted via `ast.get_source_segment`
3. **LR trained on 50 synthetic self-labeled snapshots is not credible in a research paper** → readiness scorer is now a pure rule-based weighted rubric; no joblib model, no train script
